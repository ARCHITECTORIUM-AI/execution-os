Название:
EXAMPLE_016 — Failure Protocol

Решение:
Зафиксировать протокол отказа (Failure Protocol) как обязательный порядок действий
при срыве исполнения в Execution OS (LIVE MODE).

Определение Failure (отказ):
Failure фиксируется, если выполняется хотя бы одно:
— задача не может быть завершена по Definition of DONE из-за блокера;
— задача застряла в ACTIVE без прогресса дольше допустимого окна;
— обнаружена ошибка/дефект, делающий результат неприемлемым;
— входные данные/зависимости отсутствуют или недостоверны;
— выполнение нарушает правила Execution OS (контур/приоритет/лимит).

Классы отказов:
F1 — Blocker (внешняя зависимость / ожидание / доступ / человек / ресурс)
F2 — Scope Break (для завершения требуется расширение объёма работы)
F3 — Quality Fail (результат не проходит Definition of DONE)
F4 — Process Violation (нарушены правила Execution OS)
F5 — Unclear Target (цель/критерии неполные, невозможно завершить честно)

Принцип реакции:
Failure не лечится в моменте.
Failure фиксируется как факт и переводит задачу в контролируемое состояние.

Единый алгоритм Failure Protocol:

1) STOP
— выполнение задачи останавливается;
— запрещены попытки “дожать” или “починить на ходу”.

2) TRACE
— фиксируется запись в Execution Trace:
  • Task ID / Название
  • Failure Class (F1–F5)
  • Timestamp
  • Краткое описание факта (без интерпретаций)
  • Что уже сделано (если сделано)
  • Что требуется для продолжения (если известно)

3) STATE
— задача переводится в один из статусов:
  • WAIT — если F1 (ожидание блокера)
  • HOLD — если F2/F5 (нужна перепостановка или уточнение)
  • REWORK — если F3 (нужна доработка в рамках того же объёма)
  • ESCALATE — если F4 (нарушение контура / требуется решение)

4) ROUTE
— дальнейшие решения принимаются только через:
  • Execution Review Loop (по регламенту),
  • либо через явное решение вне LIVE MODE (если требуется новая версия).

Запреты:
— исправлять Failure “на эмоциях”;
— менять приоритет, чтобы скрыть Failure;
— расширять объём работы без оформления;
— считать частично сделанное “почти DONE”.

Правила по классам:

F1 — Blocker
— задача = WAIT;
— фиксируется зависимость и владелец блокера;
— при снятии блокера задача возвращается в Queue без изменения правил.

F2 — Scope Break
— задача = HOLD;
— требуется новая постановка или новая задача (вход через Execution Inbox);
— в LIVE MODE объём не расширяется.

F3 — Quality Fail
— задача = REWORK;
— допускается доработка только в рамках текущего объёма;
— если для качества нужен новый объём → перевод в F2.

F4 — Process Violation
— задача = ESCALATE;
— фиксируется нарушение;
— исполнение не продолжается до решения в Review Loop.

F5 — Unclear Target
— задача = HOLD;
— требуется уточнение критериев DONE;
— если уточнение меняет смысл/объём → F2.

Изменение Failure Protocol:
— возможно только через Execution Review Loop;
— оформляется как новая версия Execution OS;
— в LIVE MODE запрещено.

Принцип:
Execution OS не героична.
Execution OS фиксирует реальность.
Execution OS сохраняет контур.
