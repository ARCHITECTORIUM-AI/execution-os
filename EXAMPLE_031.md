# EXAMPLE_031 — Canonical Execution Selection Protocol

Статус: ACTIVE  
Уровень: Execution OS · Core  
Тип: Selection Protocol  
Применимость: Первый Execution в рамках новой Execution OS  
Изменения: только через Execution Review Loop  
LIVE MODE: запрещено

---

## 1. PURPOSE

Canonical Execution Selection Protocol определяет порядок и критерии выбора
первого Execution в Execution OS.

Цель протокола:
— исключить случайный, удобный или преждевременный Execution;
— зафиксировать единственный допустимый способ входа Execution OS в реальность;
— защитить Execution OS от имитации действия.

Execution начинается не с желания действовать, а с обнаружения неизбежности.

---

## 2. SCOPE

Протокол применяется:
— только к первому Execution новой Execution OS;
— только один раз;
— не переиспользуется для последующих Execution.

Если Canonical Execution не выбран,
Execution OS считается архитектурно незапущенной.

---

## 3. DEFINITION: CANONICAL EXECUTION

Canonical Execution — это Execution, который:

— уже существует в реальности;
— не создан специально под Execution OS;
— не может быть отложен без потери или искажения результата;
— будет выполнен хуже без формы Execution OS.

Canonical Execution не выбирается.
Canonical Execution обнаруживается.

---

## 4. MANDATORY SELECTION CRITERIA

Кейс допускается к рассмотрению как Canonical Execution
только при одновременном выполнении всех условий ниже.

1. Реальный внешний результат  
   Execution должен приводить к изменению вне системы мышления,
   документации или архитектуры.

2. Необратимый DONE  
   Результат Execution не может быть отменён, переписан или «отыгран назад».

3. Давление времени  
   Execution имеет естественный временной контур
   и теряет ценность при откладывании.

4. Психологическое сопротивление  
   Execution вызывает внутреннее напряжение,
   сопротивление или желание заменить его другим.

5. Отсутствие инструментальной зависимости  
   Execution может быть выполнен без разработки новых инструментов,
   автоматизаций или AI-слоёв.

6. Мультиконтурное воздействие  
   Execution затрагивает не менее четырёх контуров Execution OS
   (время, ответственность, стратегия, коммуникация, реальность, риск).

Невыполнение любого пункта = автоматический REJECT.

---

## 5. HARD REJECTION CRITERIA

Кейс отклоняется без обсуждения, если:

— он улучшает систему, а не действует в реальности;
— он комфортен;
— его можно заменить аналогом без потери смысла;
— он не оставляет следа;
— он выглядит «логичным первым шагом»;
— он воспринимается как подготовка, а не как действие.

---

## 6. SELECTION AUTHORITY

Право фиксации Canonical Execution принадлежит:

— Execution Owner.

Запрещено:
— делегировать выбор;
— принимать решение коллективно;
— использовать голосование;
— поручать выбор AI.

AI допускается только как аналитический инструмент,
не как субъект решения.

---

## 7. DECISION FINALITY

После фиксации Canonical Execution:

— кейс не заменяется;
— не оптимизируется;
— не «улучшается»;
— не откладывается ради альтернатив.

Возможны только два состояния:
— Execution стартует;
— Execution OS признаётся неготовой к реальности.

---

## 8. ARTIFACT

Результатом применения протокола является:

— один зафиксированный Canonical Execution;
— отдельный Execution Slot;
— переход к Phase 2 — Execution in Practice.

---

## 9. PRINCIPLE

Execution OS не выбирает удобство.
Execution OS выбирает неизбежность.
Execution OS входит в реальность только по форме.
